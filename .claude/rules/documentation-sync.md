# 文档同步规则

每次修改代码后，**必须主动检查并更新受影响的文档和注释**，禁止等用户提醒。

## 文档分层策略

文档按"离代码距离"分为两层：

| 层级 | 存储位置 | 内容类型 | 过期风险 |
|------|---------|---------|---------|
| **代码注释** | 源码文件顶部 `@module` + JSDoc | 文件职责、函数签名、参数说明、依赖关系、消费者 | 最低（改代码时自然会看到） |
| **独立文档** | `docs/` 目录 | 跨模块交互图、架构决策（Why）、配置迁移表、开发规范/踩坑 | 中（需主动同步） |

**实现细节优先写入注释**，独立文档只保留"读单文件时看不到"的信息。

## 必须检查的场景

| 场景 | 更新范围 |
|------|----------|
| 文件重命名/移动/删除/拆分 | 所有引用路径（`grep` 旧路径）、`docs/` 文件索引、`CLAUDE.md` |
| 模块职责变更 | 对应 `docs/` 文档 + 源码顶部 `@module` 注释 |
| 新增/修改/删除函数或公共 API | 源码顶部注释（独立文档仅在有架构影响时才更新） |
| 修改配置结构（字段增删改） | `config.md` 迁移表 + `update.ts` 顶部注释 |
| 新增/修改陷阱或易错点 | `.claude/rules/pitfalls.md` |
| 架构偏离（依赖关系变化、新模块） | 对应 `docs/` 架构图 + `CLAUDE.md` 目录结构表 |
| 编码风格/约定的变更 | `development.md` |

## 检查清单

每次修改代码后，快速过一遍：
1. `grep` 旧路径/旧函数名，确认无遗漏引用
2. 修改的模块是否有 `@module` 顶部注释需要更新？
3. `docs/` 中是否有文档提及该模块或函数？（优先查对应文档索引表）
4. `CLAUDE.md` 的目录结构表是否需要更新？
5. 是否引入了新的已知陷阱需要记录到 `pitfalls.md`？
