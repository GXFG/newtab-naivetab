# 编码行为准则

> 以下准则倾向于谨慎而非速度。对于琐碎任务可自行权衡。

## 1. 编码前先思考

不要妄下断言，不要掩饰困惑。坦诚地权衡利弊。

- **明确假设**：实施前陈述假设，如有疑问立即提出。
- **消除歧义**：存在多种解释时提出来，不要默默做选择。
- **坚持简单方案**：如果有更简单的做法，提出来，必要时要坚持己见。
- **遇到困惑就停**：有不清楚的地方停下来，说出让你困惑的地方，然后提问。

## 2. 简单至上

用最少代码解决问题，不做任何推测。

- 没有超出需求的功能。
- 不为一次性代码做抽象。
- 不提供未要求的"灵活性"或"可配置性"。
- 对不可能出现的情况不做错误处理。
- 如果写了 200 行但 50 行就能完成，重写。问自己："资深工程师会认为这过于复杂吗？"

## 3. 手术式改动

只碰必须碰的东西，只收拾自己的烂摊子。

- 不"改进"相邻代码、注释或格式。
- 不重构没有问题的代码。
- 保持与现有风格一致，即使你的做法不同。
- 发现无关死代码时指出来，不要直接删除。
- 修改后删除不再使用的导入/变量/函数。
- 除非被要求，否则不要删除已有的无效代码。
- **每一行修改后的代码都应该直接追溯到用户的请求。**

## 4. 目标驱动型执行

将任务转化为可验证的目标，循环直至验证通过。

- "添加验证" → "编写针对无效输入的测试，并确保通过"
- "修复漏洞" → "编写重现漏洞的测试，然后使其通过"
- "重构 X" → "确保重构前后测试均通过"

多步骤任务简要说明计划：
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]

明确的成功标准让你能独立迭代。模糊标准（"只要能行就行"）则需要不断澄清。
