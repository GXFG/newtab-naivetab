# 工作流规则

- **改代码后立即同步文档/注释/单测（硬规则）：** 每次代码改动完成后、标记任务完成前，**必须**走一遍 [documentation-sync.md](documentation-sync.md) 检查清单。文件移动/重命名/拆分后必须全局搜索旧路径并更新所有引用（源码、单测、文档、CLAUDE.md）。**这条最容易忘，先执行再标记完成。**
- **不启动 dev server：** 浏览器扩展 CLI 环境无法查看页面，UI 改动 type-check 通过即可，验证由用户负责。
- **不改版本号：** `package.json` 的 version 只能由用户手动修改。
- **不重复造轮子：** 必须使用`src/components`、Naive UI 已有组件。
- **不留历史包袱：** 重构时直接更新所有消费者，禁止通过 re-export 保持"向后兼容"。旧路径及时消除，避免同一事物有多条 import 路径。
- **CLAUDE.md 维护：** 新规律/约定/坑点及时补充，过时描述及时修正。
- **memory 存储位置：** 重要规范类记忆写入 `CLAUDE.md`、`.claude/rules/`。
- **问题分析前置：** 每次修改问题前，必须仔细分析**是否真的是问题**。如果是则修复；如果不是（设计如此、框架特性、有意为之等），则在代码中添加清晰注释说明原因，避免下次误以为是 bug。过程中不确定的点与用户确认。修复非 bug 类"问题"后，**必须更新 `docs/` 下对应文档**以补充设计说明。
- **CSS 输出前检查：** 每次写 CSS/`<style>` 时，必须快速过一遍 [development.md](docs/conventions/development.md#编码风格) 的关键规则，禁止被既有不合规代码"带偏"：
  - BEM 子元素必须**嵌套**在父级选择器内，不平铺到根级
  - `&` 只用于伪类/伪元素（`&:hover`）或嵌套完整类名（`&.parent-class`），禁止 `&--modifier` 拼接
  - Modifier 必须写完整类名
  - `rgba()` 的 alpha 通道写**字面量**，禁止用 `var()`
  - 禁止 `v-bind()`，使用 `:style` + `computed` + `var()`
- **代码修改收尾检查：** 见上方第一条硬规则，改代码后立即同步单测、注释、文档，不得等用户提醒。
